BACKGROUND
----------
Evaluation run (limit=3, Italian, 2026-05-12) returned Faithfulness=0.504,
suggesting the LLM often adds facts not grounded in retrieved context.
Hypothesis: the current text extractor (trafilatura) may lose important
content on JS-heavy or dynamically loaded pages, causing retrieval gaps
that force the LLM to hallucinate.

Comparison methodology:
  Level A — Same HTML (requests fetch), different extraction only
             → isolates extractor quality from fetching method
  Level B — Playwright (Crawl4AI) vs requests fetch
             → measures JS-rendering benefit

Both methods configured with equivalent filters (see compare_crawlers.py
module docstring for full equivalence table). One unavoidable gap:
trafilatura uses its own ML classifier (favor_precision=True), while
Crawl4AI uses PruningContentFilter(threshold=0.48, fixed).

KEY OBSERVATIONS (Level A)
---------------------------
1. CRITICAL LOSS on JS-populated pages: trafilatura falls back to _bs4_extractor
   when its ML classifier discards main content (e.g., homepage=70 chars,
   offerta-formativa=91 chars). Crawl4AI correctly extracts 1174 and 1401 chars
   respectively. These are ZERO-information failures for RAG.

2. On content-rich static pages (e.g., presentazione=9143 chars), trafilatura
   performs comparably. The 40% average gap is driven by the JS-heavy failures.

3. Crawl4AI provides markdown structure (avg 7.6 headers per page) which helps
   the LLM orient itself in long documents and may improve faithfulness.

4. Noise ratio is slightly higher for Crawl4AI (+0.031 absolute) but this is
   partially explained by including more boilerplate from footer/nav fragments
   even after filtering. The PruningContentFilter is less aggressive than
   trafilatura's ML classifier.

5. Keyword density slightly lower for Crawl4AI (0.035 vs 0.040) because longer
   texts dilute the domain keywords — not a quality degradation.

CONCLUSION (Level A): Crawl4AI extraction is clearly superior on pages where
trafilatura's precision filter drops main content. The noise penalty is small
and acceptable. The fact preservation advantage is significant.

4. CONCLUSION & RECOMMENDATION
================================================================================

PARTIAL (based on Level A only, pending Levels B and retrieval test)
----------------------------------------------------------------------
Strong case for migrating trafilatura extraction to Crawl4AI:

Evidence FOR migration:
  [+] 39% more text extracted on average (driven by JS-page failures in traf)
  [+] Critical zero-content failures on homepage and offerta-formativa (key pages)
  [+] 100% fact recall by Crawl4AI vs 7%-89% by trafilatura depending on page
  [+] Structural markdown (7.6 headers/page) may improve LLM faithfulness
  [+] No worse on static pages (presentazione: comparable quality)

Evidence AGAINST migration:
  [-] Slightly higher noise ratio (+0.031 absolute)
  [-] Slightly lower keyword density (dilution effect, not actual quality drop)
  [-] Crawl4AI is a heavier dependency (Playwright browsers required)
  [-] Level A only: all tests use static HTML — no JS rendering tested yet

DECISION PENDING:
  Level B will determine whether to migrate the FETCHER (RecursiveUrlLoader →
  AsyncWebCrawler) in addition to the extractor.

  If Level B shows minimal diff (content_diff_ratio > 0.85) →
    Migrate extraction only (Phase 2): replace html_extractor() with Crawl4AI

  If Level B shows significant Playwright content gain (diff_ratio < 0.7) →
    Migrate both fetcher + extractor (Phase 2 + Phase 3)

  Retrieval test will confirm whether extraction improvements translate to
  measurable RAG quality gains before committing to production changes.
